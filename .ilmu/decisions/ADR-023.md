# ADR-023: Cross-File $ref Resolution Strategy

**Date**: 2026-04-15
**Status**: Accepted

## Context

ADR-020 deferred cross-file `$ref` resolution to v0.4.0. ADR-015 provides document-internal resolution (`resolve_ref` in `src/rules/util.rs`). This ADR defines v0.4.0 strategy for resolving file-relative `$ref` paths so rules operate on fully-dereferenced specs.

**Cross-file `$ref` resolution requires:**

- Resolver: given base file path + `$ref` string like `./shared/parameters.yaml#/PetId`, loads/parses target file, navigates JSON Pointer fragment, inlines result.
- Cycle detection: spec A refs spec B refs spec A.
- File-load cache: same external file may be referenced by many nodes; load once per lint run.
- Explicit rejection of HTTP `$ref` (`https://...`): refract is local CI linter, no network I/O.
- Correct path semantics: resolve relative to referencing file's directory, handle `..`, normalize across OS path separators.

**Strategy: eager pre-pass vs lazy per-rule:**

Eager pre-pass: before rules run, resolve entire document graph into single fully-inlined `serde_json::Value`. Rules receive resolved doc, see no `$ref` nodes (except unresolvable ones, which become error markers).

Lazy per-rule: rules call resolver on demand, cache shared via `LintContext`. Each rule manages own resolution needs.

Eager pre-pass wins:
- Rules stay simple. No resolver handle in `LintContext` needed for schema-validation rules.
- `oas3-schema` and `oas2-schema` require fully-resolved doc; lazy resolution doesn't help.
- Single pre-pass easier to test in isolation (pure function: `Value -> Result<Value>`).
- Failure modes centralized: missing file surfaces as one `LintError` before any rule runs, not scattered across 32 rule invocations.

Lazy resolution's only advantage: avoid resolving unused subgraphs. For OpenAPI specs (bounded size, all paths typically walked), not meaningful.

**Impact on existing rules:**

After pre-pass, all resolvable `$ref` nodes replaced with inlined content. Resulting `Value` has no external `$ref` strings (internal `#/` refs remain, resolved by `resolve_ref` as today). Existing rules invoking `resolve_ref` continue unchanged: after pre-pass, external refs gone; internal refs still need `resolve_ref` for shared components defined inline.

**OAS 3.1 $ref siblings (known limitation):**

OAS 3.1 allows `summary` and `description` siblings on a `$ref` object that override referenced content at that location. Eager pre-pass replaces entire `$ref` object with inlined content, losing siblings. v0.4.0 documents this as known limitation: rules on pre-pass output won't see sibling overrides. Affected OAS 3.1 behavior uncommon in real specs; doesn't affect structural correctness rules. Fix if needed later: after inlining, merge siblings from original `$ref` object onto inlined node (siblings win per OAS 3.1 spec).

**Partial resolution behavior:**

When `resolve_external_refs` can't resolve a ref (file not found, pointer not found, cycle, etc.), `$ref` node left in place unchanged in partially-resolved doc. Existing rules call `resolve_ref(doc, pointer, depth)` on internal `#/` refs and return `None` for external-looking strings -- skip unresolved external nodes as today. Prevents one bad ref silencing all rule violations. Corresponding `ResolveError` reported as `LintError` before rules run so user sees which refs failed.

**Memory usage:**

`resolve_external_refs` takes ownership of parsed `Value`, returns new fully-inlined `Value`. Peak memory = original doc + inlined copy held simultaneously during traversal. OAS specs assumed small (KB to low-MB range). Negligible for typical specs. No streaming or incremental approach warranted.

**Violation path reporting:**

Pre-pass inlines content but preserves original structural paths. Violations point into resolved `Value` using JSON Pointer paths. Inlining adds no structural indirection at call site (ref node replaced by target content at same JSON Pointer location), so paths remain meaningful relative to original document structure.

## Decision

**Resolver API:** `pub fn resolve_external_refs(doc: Value, base_path: &Path) -> Result<Value, Vec<ResolveError>>` in `src/resolver.rs`. Takes ownership of parsed doc, returns new fully-resolved `Value`. Errors collected (not fail-fast): all resolve errors returned, not just first.

**Cache:** `HashMap<PathBuf, serde_json::Value>` local to `resolve_external_refs` call. Not shared across lint runs. File sizes small (KB range); re-loading cost per lint run negligible. No global cache needed.

**Algorithm:**
1. Walk `Value` tree depth-first.
2. When object node contains `"$ref"` key:
   a. If value starts with `#`, skip (internal ref, handled by `resolve_ref`).
   b. If value starts with `http://` or `https://`, return `ResolveError::HttpRefForbidden`.
   c. Otherwise, parse as `path#/pointer` (pointer optional). Resolve path relative to `base_path`. Load file (cache lookup first). Navigate pointer. Replace current node with inlined content. Recurse into inlined content using target file's directory as new base (for nested external refs).
3. Track visited `(canonical_path, json_pointer)` pairs in `HashSet`. On second visit: `ResolveError::Cycle`.
4. Depth limit: 64 total external resolution steps per lint run. On exceed: `ResolveError::DepthExceeded`.

**Failure modes -> LintError mapping:**
- `ResolveError::FileNotFound` -> `LintError::UnresolvableRef { path, ref_str }` (emitted as linting error, not rule violation)
- `ResolveError::MalformedFile` -> `LintError::UnresolvableRef`
- `ResolveError::PointerNotFound` -> `LintError::UnresolvableRef`
- `ResolveError::Cycle` -> `LintError::RefCycle { path }`
- `ResolveError::HttpRefForbidden` -> `LintError::HttpRefNotSupported { ref_str }`
- `ResolveError::DepthExceeded` -> `LintError::RefDepthExceeded`

**Integration with lint():** `resolve_external_refs` called in `src/lib.rs` `lint()` after parsing, before any rule runs. Errors prepended to result. Linting continues on partially-resolved doc (best-effort) so rules still run on resolvable portions.

**Rule trait:** No change to rule signatures. Rules receive resolved doc via `LintContext::doc`.

**LintContext extension from ADR-022:** `LintContext` holds `base_path: Option<&Path>` for rules needing source file location. No v0.4.0 rule uses it; available for future.

**`Deref'd<'a>` newtype (ADR-021 escalation trigger):** Threshold is >8 deref-dependent rule files. Existing deref-dependent rules: `array-items`, `oas3-parameter-description`, `operation-parameters`, `operation-tag-defined` (4 files). New v0.4.0 rules: `oas3-schema` and `oas2-schema` operate on fully-resolved doc via pre-pass; not deref-dependent. `oas3-valid-schema-example` and `oas2-valid-schema-example` access example values and schemas; ARE deref-dependent (2 files). Total deref-dependent in v0.4.0: 4 + 2 = 6. Threshold is >8. v0.4.0 does NOT trigger `Deref'd<'a>` newtype. Defer to v0.5.0 if count grows past 8.

## Consequences

- `src/resolver.rs` new module: ~200-300 LOC, fully testable with fixture spec trees in `tests/fixtures/`.
- Rule signatures unchanged. Developer Phase 1 = resolver + tests only; no rule changes needed.
- Partial resolution on error (best-effort lint) prevents one bad external ref silencing all rule violations. Trade-off: rules may see partially-inlined doc. Acceptable: better than silent failure.
- HTTP ref rejection explicit; users get clear error, not hang or false negative.
- Cycle detection using canonical paths handles symlinks correctly (canonicalize before inserting to HashSet).
- Windows path semantics: use `dunce::canonicalize` or `std::fs::canonicalize`. Note: `std::fs::canonicalize` on Windows returns UNC paths; `dunce` normalizes these. Add `dunce` to dependencies if resolver implemented.
- Future: if HTTP $ref ever desired (unlikely for CI linter), adds `ResolveError::HttpRefForbidden -> allow` branch without touching rest of algorithm.
