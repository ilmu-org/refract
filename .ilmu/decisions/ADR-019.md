# ADR-019: JSON Schema Validation Rules Deferred to v0.4.0

**Date**: 2026-04-14
**Status**: Accepted

## Context

Four Spectral rules require JSON Schema evaluator:

- `oas3-schema`, `oas2-schema`: validate entire OAS doc against official OAS JSON Schema (2000+ line schema for OAS 3.0, separate variants for 3.1 and 2.0).
- `oas3-valid-schema-example`, `oas2-valid-schema-example`: validate every `example` value against its declaring schema.

Rules deliver real value: catch whole bug classes (`format: int32` with example `"abc"`; request body missing required keyword) that structural rules can't. Also introduce architectural cost.

**Library options assessed:**

1. **boon** (current latest 0.6.x, MIT, pure Rust). Implements JSON Schema drafts 4, 6, 7, 2019-09, 2020-12. Pure Rust, no native deps, musl-clean. Adds ~250 KB to stripped release binary plus schema docs (OAS 3.0 schema ~60 KB JSON, OAS 3.1 ~80 KB, OAS 2.0 ~90 KB). Binary grows ~0.5 MB total. Maintained, stable API.
2. **jsonschema** (most popular crate). Pulls `fancy-regex` and `regex` for `pattern` keyword, plus `url` for `format: uri`. Dependency surface ~3x boon. Historic musl build issues. Larger binary impact.
3. **valico** (legacy). Drafts 4 only, no 2020-12. Out.
4. Hand-roll. Implementing JSON Schema 2020-12 evaluation = multi-month project. Out.

**Architectural friction beyond binary size:**

- Schema evaluation produces nested error trees (single `oas3-schema` failure on malformed doc yields dozens of leaf errors). Mapping to flat `Violation`s with sensible `path` and `message` needs dedicated translator. Spectral's translator is non-trivial; refract needs its own.
- OAS JSON Schema must bundle as compile-time asset. Choice of how (`static` byte string per OAS version, lazy-parsed; `include_bytes!` + `Lazy<Value>`; or build-script-generated module) has binary-size implications.
- `oas3-valid-schema-example` and `oas2-valid-schema-example` apply evaluation to user-defined sub-schemas, not the OAS schema. Evaluator must accept arbitrary schemas at runtime; boon supports this, but integration shape (one `boon::Schemas` registry per `lint()` call vs per rule invocation) is non-trivial.
- Error-vs-violation distinction: malformed schema in input that boon refuses to compile is neither clean lint pass nor structural violation. Reporter contract must define this state.

Doing all four rules plus four design questions in v0.3.0 doubles milestone size and concentrates risk: schema-evaluation slip blocks 17 structural rules otherwise ready.

## Decision

Defer all four schema-evaluation rules to v0.4.0. Defer boon (or alternative) dependency choice to that milestone. v0.4.0's first ADR = evaluator selection and integration shape.

v0.3.0 ships no JSON Schema dependency. v0.3.0 README and CHANGELOG must note `oas3-schema` and `*-valid-schema-example` not yet implemented; users needing them today should chain Spectral pre-check or wait for v0.4.0.

## Consequences

- v0.3.0 binary size stays ~flat vs v0.2.0.
- v0.3.0 ships in reviewable window. Schema-evaluation work isolated to v0.4.0 with full architect attention.
- Users depending on full document validation today have clear gap to plan around. Docs must be honest.
- boon assessment above = starting point for v0.4.0's first ADR. If better library appears (or OASIS-blessed JSON Schema 2020-12 reference Rust impl matures), v0.4.0 reassesses.
- Once schema validation lands, refract crosses capability threshold to replace Spectral for majority of Stoplight-style use cases. v0.4.0 earns 1.0.0 candidate label.
