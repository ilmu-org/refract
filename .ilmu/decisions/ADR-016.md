# ADR-016: mod.rs → Named Module Rename Confirms Rust 2018+ Pattern

**Date**: 2026-04-14
**Status**: Accepted

## Context

v0.1.0 out_of_scope flagged `mod.rs` → named module rename as first-commit task on v0.2.0 branch. Confirming rename mechanically correct before implementation.

## Decision

Rename pattern (applied to `parser`, `model`, `rules`, `ruleset`, `reporter`):

```
Before:                          After:
src/rules/mod.rs                 src/rules.rs
src/rules/operation_tags.rs  →   src/rules/operation_tags.rs  (unchanged)
src/rules/info_contact.rs        src/rules/info_contact.rs    (unchanged)
...
```

Rust 2018+ (and Edition 2024) supports `src/rules.rs` coexisting with `src/rules/` directory holding submodules. No change to `mod` declarations inside `src/rules.rs` (formerly `mod.rs`). No change to `src/lib.rs` module declarations (`pub mod rules;` works identically).

`src/error.rs` already named module (not `src/error/mod.rs`) — no change needed.

Rename = single mechanical commit, landed first on v0.2.0 branch before feature work. Touches five files (`mod.rs` → `<name>.rs`). `cargo build` must pass identically before and after.

## Consequences

- Removes `mod.rs` ambiguity as codebase grows (v0.2.0 adds `src/rules/util.rs` and 7+ rule files; named-module style makes editor navigation materially clearer).
- No user-visible change. No API change. No binary-size change.
- First-commit discipline: rename must not bundle with feature work. Failing test after feature commits must not be chased through rename diff.
