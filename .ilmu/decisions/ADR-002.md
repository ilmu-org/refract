# ADR-002: YAML and JSON Parsing with serde + serde_yaml + serde_json

**Date**: 2026-04-09
**Status**: Accepted

## Context

OpenAPI specs authored in YAML or JSON. Parser must:
1. Detect format from file extension or content sniffing.
2. Deserialise to generic value tree (preserve all keys, including unknown extension fields like `x-*`).
3. Retain span/location info for line-accurate violation reporting.

Candidates evaluated:

| Crate | Maintained | License | Span support | Notes |
|-------|-----------|---------|--------------|-------|
| `serde_yaml` (dtolnay, v0.9) | Yes | MIT/Apache | No native span | Deserialises to `serde_yaml::Value` with `Mapping` retaining insertion order |
| `serde_yaml` v0.9 + `marked_yaml` | `marked_yaml` unmaintained | — | Yes | Dependency risk |
| `yaml-rust2` | Yes (fork of yaml-rust) | MIT/Apache | Yes (Marker) | Low-level; would need hand-written deserialization |
| `serde_json` | Yes | MIT/Apache | No native span | Needed anyway for JSON specs |
| `libyaml-safer` | Yes | MIT | Partial | Wraps libyaml; C FFI; not static-binary-friendly |

YAML span info needed for line-accurate violation reporting. `serde_yaml::Value` doesn't expose spans. Pragmatic solution for v0.1.0: parse with `serde_yaml` for document tree, then second-pass with `yaml-rust2` (or `serde_yaml`'s internal scanner) to build position index keyed on JSON Pointer paths.

For v0.1.0 position index approach sufficient: most OAS rules fire on structural paths (e.g. `paths./foo.get.responses.200`) where path-to-line lookup table accurate enough.

## Decision

- `serde_yaml = "0.9"` for YAML deserialization to `serde_yaml::Value`.
- `serde_json = "1"` for JSON deserialization to `serde_json::Value`.
- Internal `model::PositionIndex` built during parsing: `HashMap<JsonPointer, Span>` mapping each node's JSON Pointer path to its (line, col) in source. Populated by single-pass YAML/JSON event scanner run after `serde_yaml`/`serde_json` deserialization.
- `Span` is `pub struct Span { pub line: u32, pub col: u32 }` — simple, allocation-free.

## Consequences

- Two-pass parsing adds small constant overhead (~5–10% on large specs). Acceptable.
- `serde_yaml` 0.9 uses `unsafe-libyaml` under the hood (pure Rust YAML parser). Static binary friendly, no C dependency.
- `serde_json` standard; zero risk.
- Position index covers structural nodes only. Sufficient for v0.1.0. Inline value spans (e.g. column offset within string) deferred to v0.2.0.
