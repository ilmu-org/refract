# ADR-013: Reporter API — Batch Signature for Multi-File Correctness

**Date**: 2026-04-14
**Status**: Accepted
**Supersedes**: plan.md proposed `report(violations, spec_path, format, color, out)` signature

## Context

v0.2.0 plan proposed single-file reporter signature:

```rust
pub fn report(violations: &[Violation], spec_path: &str,
              format: Format, color: ColorMode,
              out: &mut dyn Write) -> std::io::Result<()>
```

Works for text/JSON (caller loops, calls `report()` per file). **Breaks for SARIF**: SARIF log = single JSON doc, one `run`, `results` array spanning all files. Each result's `physicalLocation.artifactLocation.uri` identifies file. One SARIF doc per file = invalid SARIF (GitHub Code Scanning expects single `.sarif`).

Two options:

- **Option A**: batch signature, all file results at once. Text/JSON iterate internally; SARIF emits one doc.
- **Option B**: two entry points — `report()` per-file for text/JSON, `report_sarif()` batch for SARIF.

Option A cleaner: one function, one signature, one call site in `main.rs`. Single-file linting wraps in one-element slice. Avoids caller-side complexity of "sometimes batch, sometimes per-file" API.

## Decision

v0.2.0 reporter API:

```rust
pub enum Format { Text, Json, Sarif }
pub enum ColorMode { Auto, Always, Never }

pub fn report(
    files: &[(PathBuf, Vec<Violation>)],
    format: Format,
    color: ColorMode,
    out: &mut dyn Write,
) -> std::io::Result<()>
```

- `ColorMode::Auto` resolves enabled when `out` points at TTY **and** `NO_COLOR` unset; else disabled. Resolution inside reporter via `std::io::IsTerminal` on caller-provided stream (pragmatic limit: file handle wrapping stdout still resolves correctly — caller passes `stdout().lock()` directly).
- **Text**: per-file iteration; no file-group header (violations self-identify via `{file}:{line}:{col}` prefix); trailing summary after last file (`N files linted, M violations (E errors, W warnings)`).
- **JSON**: top-level shape becomes `{ "files": [{"file": "...", "violations": [...]}], "summary": {...} }`. Single-file invocation still produces one-element `files` array — minor v0.2.0 breaking change vs v0.1.0's `{ "violations": [...], "summary": {...} }`. Acceptable pre-1.0; document in release notes.
- **SARIF**: one doc, one `run`, `tool.driver.rules` from registered rules, `results` spanning all files, `artifacts` listing each input file.

## Consequences

- Supersedes plan's proposed signature. Plan architect notes will flag change.
- `main.rs` calls `report()` once regardless of single-file or directory-scan. Exit code logic stays in `main.rs`.
- JSON v0.1.0 → v0.2.0 minor breaking change. Format bump documented; consumers wanting v0.1.0 shape stay on v0.1.x.
- SARIF requires line/col from ADR-011 — not new dependency, but hard sequencing constraint: land ADR-011 before ADR-013.
- `write_text` / `write_json` removed (already slated for collapse per v0.1.0 out_of_scope). Internal helpers `write_text_impl`, `write_json_impl`, `write_sarif_impl` live as private functions dispatched from `report()`.
