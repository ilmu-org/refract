# ADR-020: Cross-File $ref Resolution Deferred to v0.4.0

**Date**: 2026-04-14
**Status**: Accepted

## Context

ADR-015 shipped document-internal `$ref` resolution (refs of form `#/components/...`). Cross-file `$ref` (refs of form `./shared/parameters.yaml#/PetId` or `../common.json`) out of scope for v0.2.0, reconsidered now.

**What cross-file resolution would require:**

- Resolver: given base file path + `$ref` string, loads target file (YAML/JSON), caches it, returns `serde_json::Value` + parsed JSON Pointer fragment.
- Cycle detection across file boundaries (A refs B refs A).
- Path semantics: relative resolution against current file, handles `..`, symlinks, Windows separators, case-sensitivity across filesystems.
- Loader cache shared across lint run (multiple rules/operations may deref same external file).
- Failure modes: missing file, malformed file, pointer-not-found, cycle exceeded. Each maps to lint violation (rule invariant broken) or `LintError` (document graph broken).
- HTTP `$ref` (`https://example.com/shared.yaml#/...`) out of scope at all milestones (no network in CI linter). Resolver must reject explicitly.

**Impact on v0.3.0 rules:**

17 v0.3.0 rules, 4 consume `$ref` deref. With internal-only deref:

- `array-items` on schema behind external `$ref` treated as opaque, no violation, no false positive. Acceptable: most arrays-without-items mistakes in inline schemas.
- `oas3-parameter-description` on parameter behind external `$ref` skipped. Acceptable: parameter component file can be linted directly when scanned.
- `operation-parameters` (uniqueness of (name, in)) compares deref'd parameters. External-ref'd parameters compare by `$ref` string: two operations referencing same external parameter correctly seen as identical, but two different external refs resolving to same name+in won't be caught. Acceptable: rare in practice.
- `operation-tag-defined` compares string tag values directly, no deref needed.

Degradation bounded: rules produce no false positives on external-ref'd nodes, only false negatives. Lint signal stays trustworthy.

**Why defer:**

Users most affected by external `$ref`: teams with bundled spec workflows (Redocly, Stoplight Studio, swagger-cli bundle). Those teams lint bundled output where every ref is internal. Teams linting un-bundled multi-file specs are minority. Cross-file resolution is multi-week effort for minority use case in v0.3.0; competes with 17 new rules for architect/developer attention.

Pairing cross-file `$ref` with schema validation (v0.4.0) is natural: `oas3-schema` can't validate document with unresolved external refs without resolving them first.

## Decision

Defer cross-file `$ref` resolution to v0.4.0. Internal-only `deref` from ADR-015 remains contract for v0.3.0 rules.

External `$ref` handling stays permissive in v0.3.0: when `deref` encounters non-internal `$ref` (any string not starting with `#/` or equal to `#`), returns original `{"$ref": "..."}` unchanged. Calling rule treats as opaque (no violation, no field inspection post-deref).

v0.3.0 README must document this explicitly: "Cross-file `$ref` is not resolved in v0.3.0. Bundle multi-file specs with `redocly bundle` (or equivalent) before linting for full coverage." Include v0.4.0 milestone link.

## Consequences

- No new resolver, loader cache, or filesystem-traversal code in v0.3.0. Risk surface stays small.
- Bundled-spec workflows (majority) fully covered.
- Multi-file workflows hit documented limitation, not silent failure or false positive.
- v0.4.0 owns resolver design alongside schema validation. Both need "fully resolved document" abstraction; building both same milestone lets abstraction emerge from real use, not speculation.
- Deref contract (return-unchanged on external) is now two-version-stable invariant. v0.3.0 rules need no changes when v0.4.0 lands cross-file resolution: deref will simply succeed on more inputs.
