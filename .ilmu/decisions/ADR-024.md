# ADR-024: Workspace Structure Assessment for v0.4.0

**Date**: 2026-04-15
**Status**: Accepted

## Context

ADR-001 established single-crate structure. v0.3.0 shipped unchanged. `refract-core` workspace extraction deferred twice (v0.2.0, v0.3.0) pending concrete external consumer. This ADR makes v0.4.0 decision explicit.

**Arguments for extraction:**

- Library crate lets downstream tools (custom linters, spec processors, IDE plugins) reuse refract's parsing, rule evaluation, violation types without bundling CLI.
- Workspace separation enforces API discipline: public types stable, internal types stay in `refract-cli`.
- `LintContext` (ADR-022) and `resolve_external_refs` (ADR-023) are natural library-crate inhabitants.

**Arguments against extraction in v0.4.0:**

- No external consumer filed issue requesting library access. Extracting without consumer = speculative work adding CI complexity, publishing setup, semver obligations with no payoff.
- v0.4.0 already high-complexity (resolver, boon integration, 4 new rules, LintContext refactor). Workspace extraction risks milestone slip.
- Single-crate LOC under 5,000 (ADR-001 threshold ~50K). No structural pressure.
- `refract-core` requires stable public API commitment once extracted. ADR-022's `LintContext` struct and rule trait still evolving; locking as public API premature.

**Current codebase scale:**

Single crate, ~3,000-4,000 LOC for v0.4.0 after new rules and resolver. Well within single-crate territory.

## Decision

Do not extract `refract-core` in v0.4.0. Single-crate structure (ADR-001) continues.

Next review trigger — one must occur before reconsidering:
1. Library consumer opens GitHub issue requesting crate-level access to refract's lint API.
2. refract grows secondary binary (e.g., `refract-lsp` language server, `refract-serve` HTTP API) needing shared types without CLI dependency.
3. Codebase exceeds ~15,000 LOC and module boundaries become hard to navigate.

Until trigger fires, workspace extraction deferred with no fixed milestone.

## Consequences

- v0.4.0 carries no cargo workspace setup work. Developer focuses on resolver, boon, rules.
- `LintContext`, rule trait, `Violation`, `LintError` remain `pub(crate)` or module-scoped. No semver obligation.
- When extraction happens, trigger condition provides clear motivation for API design. Extracting without consumer produces APIs for imagined use cases; extracting for real consumer produces APIs shaped by real needs.
