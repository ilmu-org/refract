# ADR-011: Position Indexing via yaml-rust2 Two-Pass Scan (v0.2.0)

**Date**: 2026-04-14
**Status**: Accepted
**Supersedes**: none (extends ADR-002 for v0.2.0)

## Context

v0.1.0 shipped path-only violations (no `line`/`col`) — two-pass scan underspecified at critic-review. v0.2.0 needs line/col for two reasons:

1. Editor integration (users clicking CI log jump to right line).
2. SARIF output (ADR-013) requires `physicalLocation.region.startLine` — cannot omit.

Candidates re-evaluated:

| Crate | Maintained | License | Span support | C FFI | Verdict |
|-------|-----------|---------|--------------|-------|---------|
| `serde_yaml` 0.9 native | Yes | MIT/Apache | No | No (uses pure-Rust `unsafe-libyaml`) | Cannot — no span API |
| `marked-yaml` | No (unmaintained) | — | Yes | No | Rejected per ADR-002 |
| `yaml-rust2` | Yes | MIT/Apache | Yes (`Marker`) | No (pure Rust) | Selected |
| `libyaml-safer` | Yes | MIT | Partial | Wraps C libyaml | Rejected — breaks musl story |
| Custom `serde_json::RawValue` post-scan | N/A | — | Partial (byte offsets only) | No | Complex; no line/col mapping |

`serde_yaml` 0.9 and `yaml-rust2` are **independent parsers that coexist cleanly**. Both in one binary: no interference, no shared global state, operate on `&str` input. `unsafe-libyaml` (pure-Rust backend of `serde_yaml`) unrelated to C `libyaml` despite name.

## Decision

YAML spec files use two-pass approach in v0.2.0:

1. **Pass 1** (existing): `serde_yaml::from_str(content)` produces `serde_yaml::Value`, normalised to `serde_json::Value` for rule engine. No change.
2. **Pass 2** (new): `yaml_rust2::parser::Parser::new_from_str(content)` drives event-based scan. Visitor maintains stack tracking:
   - Current JSON Pointer path (built from alternating key/value state inside `MappingStart`/`MappingEnd` frames, plus monotonic index inside `SequenceStart`/`SequenceEnd` frames)
   - `Marker` (line, col) attached to each event
   
   Each scalar, mapping-start, or sequence-start event: visitor records `(pointer.clone(), Span { line, col })` into `HashMap<String, Span>`.

3. `parser::parse()` returns both `serde_json::Value` document and `PositionIndex`. Signature becomes:
   ```rust
   pub fn parse(path: &Path) -> Result<(serde_json::Value, PositionIndex), LintError>
   ```
4. `lib::lint()` owns index, resolves each `Violation`'s `path` → `Span` **after** `rule.check()` returns. Rules don't know about `PositionIndex` — still emit `path: String` only.

JSON spec files in v0.2.0 ship `PositionIndex::empty()` (no line/col). SARIF and text output degrade gracefully: text drops `:line:col` suffix; SARIF emits `region.startLine = 1`. Explicitly deferred to v0.3.0 (see out_of_scope).

## Consequences

- No new unsafe code, no C FFI, no musl regression. `yaml-rust2` pure Rust, MIT/Apache.
- `yaml-rust2` adds ~+300 KB to binary. Acceptable (budget ~5 MB).
- Second-pass visitor is most complex new code in v0.2.0. Implementation must handle:
  - Key-to-value alternation inside mappings (toggle boolean as each pair completes)
  - Escape of `/` and `~` in JSON Pointer keys (RFC 6901: `~` → `~0`, `/` → `~1`)
  - Empty document (produces only `StreamStart` and `StreamEnd` events)
  - Anchor/alias resolution — alias node located where alias *appears*, not at anchor
- Ship dedicated unit tests for position indexer: nested maps, arrays of objects, JSON Pointer escaping, missing-path lookups (returns `None`).
- Rules' `check()` signature unchanged. Load-bearing: rule trait needs no span knowledge.
