# ADR-029: v1.0.0 Scope — Drop-in Spectral OAS Replacement (Declarative Configs)

**Date**: 2026-04-16
**Status**: Accepted

## Context

`state.md` defined v1.0.0 as "public launch -- promote when feature-complete + stable". Too vague: every scoping agent re-interprets "feature-complete" against own read of Spectral, milestone drifts. Each agent invents scope instead of inheriting stable shared definition.

Project needs concrete, checkable v1.0.0 definition any agent can verify candidate work against before proposing plan.

ADR produced after:
- rust-architect pass refining proposed criteria against ADR-008, ADR-009, ADR-023, ADR-026, ADR-027, current repo state.
- rust-critic pass adversarially challenging "drop-in" claim, parity corpus deferral, function library completeness, remote extends question.

Key critic findings incorporated:
- "Drop-in" must be qualified to declarative rulesets only; JS exclusion permanent and material.
- Parity corpus cannot defer to v1.0.1; only external verification of claim.
- JSONPath dialect must lock via ADR before any custom-rule implementation; dialect divergence causes silent violation-set differences from Spectral, unit tests don't catch.
- Remote `extends` URLs permanently forbidden by same reasoning as HTTP `$ref` in ADR-023, not deferred.
- Unknown-key policy in `.spectral.yaml` parsing needs explicit decision.

## Decision

**v1.0.0 end-goal:** drop-in replacement for Spectral OAS linting in teams using declarative `.spectral.yaml` configs. Team running `spectral lint spec.yaml` can replace binary with `refract spec.yaml`, keep existing `.spectral.yaml` unchanged (including custom declarative rules using `given`/`then.function`/`message` and any built-in Spectral functions), see equivalent violations. Parity verified by corpus of real-world specs run through both tools with equivalence asserted on rule ID, severity, message text.

**Permanent exclusions:**
- JavaScript custom function rules. refract executes rules as native Rust. JS execution impossible without abandoning single-static-binary constraint. Not deferred; will not ship.
- `spectral:asyncapi` ruleset. Out of scope v1.0.0. Track separately for v1.x if demanded.
- Remote `extends` URL resolution. Permanently forbidden by same reasoning as HTTP `$ref` (ADR-023): refract is local CI linter, no network I/O, CI reproducibility. Remote extends adds supply-chain risk (executable policy from network at lint time). Emits `LintError::HttpExtendsNotSupported`. Users vendor rulesets locally.

**Scope of work required** captured in `.ilmu/v1-readiness.md` as tiered checklist. Agents must read that document before scoping v0.6.0 onward.

## Consequences

- Future scoping agents gate candidate work against `.ilmu/v1-readiness.md`. Work not addressing checkbox without acknowledged v1.x deferral requires explicit justification.
- Teams on JS function rules get documented incompatibility + migration guidance in README: port to declarative form if possible; keep Spectral for that ruleset if not.
- AsyncAPI users directed to Spectral until v1.x or later scoped.
- Remote extends users directed to vendor rulesets locally.
- "Drop-in" claim explicitly qualified: drop-in for declarative configs, not JS-function rulesets. README "When to use what" section must reflect this before v1.0.0 ships.
- v1.0.0 depends on v0.6.0 shipping `oas3-unused-component` with correct reference graph (ADR-027 prerequisites). If v0.6.0 slips, v1.0.0 slips or item explicitly removed from Tier 1.
