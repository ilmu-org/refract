# ADR-001: Single-Crate Structure for v0.1.0

**Date**: 2026-04-09
**Status**: Accepted

## Context

openapi-linter ships as single binary. No library consumer outside binary in this milestone. Workspace adds indirection (virtual manifest, cross-crate dependency resolution, publish sequencing) without value at this scale.

## Decision

Single crate with internal module boundaries. `src/lib.rs` owns all business logic. `src/main.rs` thin entry point calls into lib. Modules:

```
src/
  main.rs          — entry point, process exit codes
  lib.rs           — public API surface (parse, lint, format output)
  parser/          — OpenAPI document loading (YAML + JSON → internal model)
  model/           — internal OpenAPI document representation
  rules/           — built-in OAS ruleset, rule trait, rule registry
  ruleset/         — .spectral.yaml file loading and merging
  reporter/        — violation formatting (text, JSON)
  error.rs         — crate-level error types (thiserror)
```

## Consequences

- Simple build: `cargo build --release` produces binary directly.
- No cross-crate API surface to maintain — all types `pub(crate)` by default.
- If downstream library consumer emerges (v0.2.0+), extracting `openapi-linter-core` crate is straightforward refactor — module boundaries already match crate boundaries.
- Slight risk: codebase past ~25K LOC may feel cramped at module level. Acceptable for v0.1.0.
