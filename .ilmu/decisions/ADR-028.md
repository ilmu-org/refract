# ADR-028: Deref'd<'a> Newtype Deferred Again Despite >8 Trigger

**Date**: 2026-04-16
**Status**: Accepted

## Context

ADR-021 established deref-before-compare as doc/review contract, not type-system enforcement. Rejected `Deref'd<'a>(&'a Value)` newtype in v0.3.0: "marginal safety for 4 rules." Explicit revisit trigger: "Revisit if v0.4.0 pushes deref-dependent rule count above 8."

**Current count after v0.5.0:**

Deref-dependent rules at end of v0.4.0: 7 (array_items, oas3_parameter_description, operation_parameters, operation_tag_defined, oas3_valid_schema_example, oas2_valid_schema_example, path_params). ADR-021 authored at count 4, projected 6 for v0.4.0, undercounted path_params from v0.1.0.

v0.5.0 adds 3 deref-dependent rules (ADR-026): oas2-parameter-description, oas3-valid-media-example, oas2-valid-media-example.

Total: 10. Exceeds ADR-021's >8 trigger.

**Evidence of bugs from missing deref:**

Zero reported false positives attributable to missing deref across v0.3.0 and v0.4.0 release cycles. All 7 existing deref-dependent rules passed review and tests. Comment contract in `src/rules/util.rs` + PR template reviewer checklist enforce invariant by discipline.

**Cost of refactor if adopted now:**

- Change signature of `resolve_ref` to return `Option<Deref<'a>>`.
- All 7 existing rule files plus util.rs update consumer signatures. Each rule's comparison helpers take `Deref<'a>` instead of `&Value`.
- Add `Deref<'a>` type, `Deref::get()` accessor, `Deref::from_unchecked()` escape hatch for walkers needing root doc. Escape hatch weakens type-system guarantee.
- Touches every new v0.5.0 rule, expanding milestone surface area.
- No observed bug being prevented.

**Trigger re-evaluation:**

ADR-021's ">8" was speculative — anticipated deref-dependent rules reaching deeper into $ref chains where silent miscompare would surface as FP schema errors. v0.4.0 shipped those rules. No such FP class emerged. Trigger fired; predicted bug class did not.

## Decision

Override ADR-021's >8 trigger. Do not introduce `Deref'd<'a>` newtype in v0.5.0.

**New trigger: revisit when any fires:**

1. Release-blocking FP bug traced to missed `deref` call.
2. Deref-dependent rule count reaches 15 (doubles current).
3. `LintContext` grows to require deref-before-compare at context level (shared cache, global deref state).

**Periodic review (mandatory):**

While deref-dependent count >= 10, every milestone scoping must revisit this ADR: re-examine new deref-adjacent bugs, re-evaluate adoption cost, confirm deferral holds. Count-15 trigger doesn't substitute for this review; it's hard ceiling if review skipped. Scoping agents: check ADR-028 status, document "reviewed, no change" or escalate.

Until then, doc and review enforce invariant.

**Status of ADR-021:**

ADR-021 invariant (deref-before-compare) remains in force. PR template checklist stays. `util.rs` doc-comment contract unchanged. This ADR overrides only the ">8 triggers newtype" clause.

## Consequences

- v0.5.0 carries no typesystem refactor. All 6 new rules consume `&Value`, call `resolve_ref` directly.
- Review discipline is durable line of defence. Missed-deref bug in v0.5.0 or later reopens ADR.
- `resolve_ref` signature stays `fn(doc: &Value, ref_str: &str, depth: usize) -> Option<&Value>`. No churn for consumers.
- Future missed-deref bug found in audit: evidence replaces speculation, newtype ships with clear justification.
- ADR-021 partially superseded (trigger number only). Index lists both; future architects read ADR-021 for invariant, ADR-028 for escalation policy.
