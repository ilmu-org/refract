# ADR-021: OAS-Version Gating Helper and Deref-Before-Compare Invariant

**Date**: 2026-04-14
**Status**: Accepted

## Context

v0.3.0 introduces two cross-cutting structural concerns spanning multiple rules, settled once not per rule.

**Concern 1: OAS-version gating.** Five v0.3.0 rules apply only to OAS 3.x specs (oas3-api-servers, oas3-server-trailing-slash, oas3-server-not-example.com, oas3-parameter-description, plus strict form of no-$ref-siblings). Existing v0.2.0 rules navigate 2.x vs 3.x shape difference internally per rule — works but duplicates detection logic. Adding 5 more 3.x-only rules without shared helper compounds duplication.

**Concern 2: Deref-before-compare.** Four v0.3.0 rules (array-items, oas3-parameter-description, operation-parameters, operation-tag-defined) cross-reference nodes that may be `$ref` objects. Correctness invariant: every comparison or field access on potentially-ref'd node must call `deref` first. Forgetting deref produces false positives. Single most likely correctness bug class in v0.3.0.

**Concern 3 (related): typed-enum coercion semantics.** `typed-enum` checks each enum value matches declared `type`. JSON has six primitive types (string, number, integer, boolean, null, array, object). Strict comparison rejects `enum: [1, 2, 3]` under `type: integer` because `serde_json::Value::Number` not split on integer-vs-float. Rule needs documented coercion semantics or ships broken.

## Decision

**Version gating helper.** Add small helper to `src/rules/util.rs`:

```rust
pub(crate) enum OasVersion {
    V2,
    V3_0,
    V3_1,
    Unknown,
}

pub(crate) fn detect_oas_version(doc: &serde_json::Value) -> OasVersion;
```

Detection logic: if `doc.swagger == "2.0"`, return `V2`. If `doc.openapi` starts with `"3.0"`, return `V3_0`. If `doc.openapi` starts with `"3.1"`, return `V3_1`. Otherwise `Unknown`.

OAS 3.x-only rules guard `check()` body with `matches!(detect_oas_version(doc), OasVersion::V3_0 | OasVersion::V3_1)`, return early on mismatch. Existing v0.2.0 rules not refactored in v0.3.0 (separate cleanup PR if appetite exists).

Rule trait stays unchanged. Version gating is per-rule guard, not registration-time filter — some rules apply to both versions with version-specific branches inside.

**Deref-before-compare invariant.** Codify in `src/rules/util.rs` as doc-comment contract on `deref`:

> Any rule that cross-references a node which may legally be a `$ref` object MUST call `deref` before reading non-`$ref` fields or comparing the node against another. Skipping `deref` produces false positives on every well-structured spec.

Each of four affected rules carries inline comment identifying it as deref-dependent, links to ADR-021. Reviewer checklist (in PR template) gains: "Does this rule access fields on a node that could be a `$ref`? If yes, is `deref` called first?"

No type-system enforcement in v0.3.0. `Deref'd<'a>(&'a Value)` newtype considered, rejected: forces wrapper through entire rule body for marginal safety, complicates existing rule signatures, invariant enforceable by review for 4 rules. Revisit if v0.4.0 (cross-file `$ref` and schema evaluation) pushes deref-dependent rule count above 8.

**typed-enum coercion semantics.** `typed-enum` treats `type: integer` and `type: number` as compatible with any `serde_json::Value::Number`, plus one extra check for `type: integer`: number must satisfy `n.is_i64() || n.is_u64() || (n.as_f64().map_or(false, |f| f.fract() == 0.0))`. Rationale: YAML and JSON both round-trip integers as numbers; rejecting `1.0` under `type: integer` surprises users. Matches Spectral's lenient behaviour, avoids JSON-vs-YAML representation gotchas.

`type: string` matches `Value::String`. `type: boolean` matches `Value::Bool`. `type: null` matches `Value::Null`. `type: array` matches `Value::Array`. `type: object` matches `Value::Object`. Multi-type (`type: ["string", "null"]`, OAS 3.1) passes if any listed type matches.

## Consequences

- 5 version-gated rules share one detection path. Future OAS 3.2 (when ratified) extends `OasVersion`, rules opt in by pattern match.
- Deref-before-compare contract documented in code, ADR, PR template. No type-system cost. Re-evaluated if rule count grows.
- typed-enum lands with documented coercion semantics, no surprise rejection of integer-shaped floats. Test fixtures cover `[1, 2, 3]` and `[1.0, 2.0]` under `type: integer`, plus mixed-type and OAS 3.1 multi-type cases.
- `src/rules/util.rs` grows from one (`deref`, `resolve_internal_ref`) to two concerns (add `detect_oas_version`, `OasVersion`). Still single-file. If util.rs exceeds ~300 lines, split in v0.4.0 along concern boundaries.
