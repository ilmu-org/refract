# ADR-022: Boon Crate Integration for JSON Schema Validation

**Date**: 2026-04-15
**Status**: Accepted

## Context

ADR-019 deferred JSON Schema validation to v0.4.0, named boon (MIT, pure Rust) as leading candidate. Four rules depend on it: `oas3-schema`, `oas2-schema`, `oas3-valid-schema-example`, `oas2-valid-schema-example`. This ADR makes final library decision and defines integration surface.

**Library decision -- boon 0.6.1 confirmed:**

Boon 0.6.1 implements JSON Schema drafts 4, 6, 7, 2019-09, 2020-12. Pure Rust, no native deps, musl-clean. API: `boon::Compiler` builds `boon::Schemas` registry; `Schemas::validate(instance, schema_url)` returns `ValidationError` (iterator over output units). API stable across 0.5 -> 0.6, no breaking changes to validate surface.

`jsonschema` crate (0.46.0) pulls `fancy-regex` and `regex` (large compile-time cost), `url`, `percent-encoding`. Dep surface ~3x boon. Historic musl build failures upstream. Adds ~1.5 MB to stripped binary vs boon's ~0.25 MB. Rejected.

**Binary size estimate (revised):**

Boon in musl binary: ~250 KB stripped. OAS JSON Schema assets as compile-time `&str` constants: OAS 3.0 ~60 KB, OAS 3.1 ~80 KB, OAS 2.0 ~90 KB. Total asset cost: ~230 KB. Net binary growth: ~0.5 MB. Confirms ADR-019 projection. Stays within 8 MB budget (~5.5 MB after boon).

**Integration shape decisions:**

1. **Registry lifetime**: One `boon::Schemas` per `lint_dir()` invocation, shared across all files. For single-file `lint()`, one registry per call. OAS JSON Schema pre-registered once at construction. `oas3-valid-schema-example` and `oas2-valid-schema-example` add user-defined schemas to same registry during `check()`. Registry passed via `LintContext`. Per-file registry in directory scan recompiles OAS JSON Schema at O(file-count) cost; sharing avoids this.

2. **Schema bundling**: OAS JSON Schema files bundled via `include_str!()` as `static` string constants. Parsed lazily into `serde_json::Value` on first use via `std::sync::OnceLock`. One `OnceLock` per OAS version (three total). No runtime file I/O, schema loading off hot path after first call.

3. **Error translation**: `boon::ValidationError` yields output units via iteration. Per leaf unit: emit one `Violation` with `rule_id` = rule name, `path` = JSON Pointer from unit's instance location, `message` = unit's error description. Non-leaf units skipped. Depth limit: tree exceeds 64 leaf violations per rule invocation → truncate, append synthetic violation: `"... N more schema violations omitted"`. Prevents unbounded output on malformed docs.

4. **Malformed input schema in user spec**: When `oas3-valid-schema-example` or `oas2-valid-schema-example` hits schema boon refuses to compile (e.g. `type: invalid-string`), treat as `LintError` variant, not `Violation`. Schema syntactically broken; example correctness check not meaningful. Surface as `LintError::MalformedSchema { path, message }`.

5. **Rule signature impact**: Rules needing boon require `Schemas` registry access. Two options: (a) add `schemas: &boon::Schemas` param to `Rule::check()`; (b) wrap in `LintContext` passed alongside `doc` and `version`. Option (b) cleaner for future extension. ADR-022 adopts `LintContext`; existing rules receive it but ignore schemas field.

## Decision

Adopt boon 0.6.1. Integration surface:

- `LintContext<'a>` struct in `src/lint.rs` holding `doc: &'a Value`, `version: OasVersion`, `schemas: &'a boon::Schemas`.
- Rule trait gains `fn check(&self, ctx: &LintContext<'_>) -> Vec<Violation>`. All 32 existing rules updated to accept `LintContext` but use only `ctx.doc` and `ctx.version` (no behavior change).
- OAS JSON Schemas bundled via `include_str!()`, parsed lazily via `OnceLock<Value>` in `src/schemas.rs`.
- Error translation: one `Violation` per boon leaf output unit, truncated at 64 per rule call.
- Malformed user schema: `LintError::MalformedSchema`.

## Consequences

- Rule trait signature changes: all 32 existing rules need mechanical update (`doc: &Value, version: OasVersion` -> `ctx: &LintContext<'_>`). No behavior change; find-replace pattern. Developer handles in Phase 2.
- Binary grows ~0.5 MB. Stays within 8 MB budget.
- `LintContext` is extension point for future lint state (e.g., base file path for cross-file resolution, resolver handle). ADR-023 adds `resolver` to `LintContext`.
- boon compile time: ~10-15 seconds incremental. Acceptable.
- OAS schema `OnceLock` init synchronous, single-threaded (no async runtime per ADR-010). No lock contention risk.
