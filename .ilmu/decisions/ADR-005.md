# ADR-005: Error Handling with thiserror in lib, anyhow in main

**Date**: 2026-04-09
**Status**: Accepted

## Context

Project follows org-level Rust workspace standard: `thiserror` for lib error types, `anyhow` for app-level propagation. CLI tool with lib core in `src/lib.rs`.

## Decision

- `src/lib.rs` + all modules under `src/`: use `thiserror` for typed error enums. Errors specific and matchable.
- `src/main.rs`: use `anyhow::Result` for top-level propagation. Lib errors wrapped with `.context()`.
- No `unwrap()` or `expect()` in lib code. Panics only in test assertions.

Error types:

```rust
// src/error.rs
#[derive(Debug, thiserror::Error)]
pub enum LintError {
    #[error("cannot read spec file: {0}")]
    Io(#[from] std::io::Error),
    #[error("cannot parse YAML: {0}")]
    Yaml(#[from] serde_yaml::Error),
    #[error("cannot parse JSON: {0}")]
    Json(#[from] serde_json::Error),
    #[error("invalid OpenAPI document: {0}")]
    InvalidSpec(String),
    #[error("cannot load ruleset: {0}")]
    Ruleset(String),
}
```

## Consequences

- `thiserror` errors testable + matchable. Lib consumers (v0.2.0+) can pattern-match variants.
- `anyhow` in main: ergonomic error display, no output error types needed.
- Consistent with org-level `[workspace.dependencies]` standard.
